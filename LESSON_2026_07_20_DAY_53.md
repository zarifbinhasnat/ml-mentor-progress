## 🧠 CONCEPT OF THE DAY

**Intuition first.** A Python `dict` lookup is a *hard* lookup: you give it a key, it walks (conceptually) to the one bucket whose key matches and hands you back exactly that value — everything else in the dict is invisible to the query, full stop. Attention is what you get if you make that lookup **soft**. Instead of "find the one key that matches and return its value," you compute *how well the query matches every key*, turn those match-scores into a probability distribution (they're non-negative and sum to 1), and return a **weighted blend of every value**, weighted by how relevant its key was. Nothing is discarded the way `dict.get()` discards every non-matching bucket — a key that's only "sort of" relevant still contributes a little; the best-matching key just contributes the most.

**Then the math.** Given a query vector $q$ and a set of $n$ key vectors $k_1, \dots, k_n$ (each paired with a value vector $v_1, \dots, v_n$):

Similarity score between the query and each key:

$$e_i = q \cdot k_i$$

Turn scores into weights that are non-negative and sum to 1:

$$w_i = \frac{\exp(e_i)}{\sum_{j=1}^{n} \exp(e_j)} = \text{softmax}(e)_i$$

The soft-lookup output — a weighted blend of every value, not a single winner:

$$\text{output} = \sum_{i=1}^{n} w_i \, v_i$$

That's the entire mechanism. $q$, $k_i$, $v_i$ are just names for now — where they actually *come from* in a real network (projections of the decoder state and the encoder hidden states) is tomorrow's concept. Today is purely: given some query and some keys/values, how do you combine them?

**Why it matters / where it leads.** Map this straight onto yesterday's diagnosis: the decoder is the query, the encoder's hidden states $h_1^{\text{enc}}, \dots, h_{T_x}^{\text{enc}}$ are simultaneously the keys *and* the values, and the bottleneck vector $c$ from Day 52 gets replaced by $\text{output} = \sum_i w_i h_i^{\text{enc}}$ — recomputed fresh at *every decoder step*, so different output words can attend to different source positions. The fixed-size funnel is gone: the decoder now has access to all $T_x$ hidden states, and it learns, via the softmax weights, which ones matter for the word it's currently generating. Tomorrow (concept 54) formalizes this with the actual $Q$, $K$, $V$ terminology and shows where each one comes from as a learned linear projection — today's job was to build the mental model of "soft, differentiable dictionary lookup" so that formalism has somewhere to land.

## 🐍 PYTHONIC EDGE

The bad way computes attention with an explicit Python loop over keys — it works, but it's slow and it's not how any real implementation does it. The clean way is a handful of vectorized ops.

```python
import torch
import torch.nn as nn
import torch.nn.functional as F

torch.manual_seed(42)
n_keys, d = 8, 4
keys = torch.randn(n_keys, d)     # shape (8, 4) -- 8 key vectors
values = torch.randn(n_keys, 4)   # shape (8, 4) -- 8 value vectors, paired 1:1 with keys
query = torch.randn(d)            # shape (4,)   -- a single query vector

# --- BAD: explicit Python loop, O(n) round-trips into/out of the interpreter ---
scores_slow = []
for i in range(n_keys):
    scores_slow.append(torch.dot(query, keys[i]))  # .dot() needs two 1-D tensors
scores_slow = torch.stack(scores_slow)              # stack: list[Tensor] -> Tensor, new dim 0

# --- GOOD: one matmul, no Python-level loop at all ---
scores_fast = keys @ query
# `@` is matmul (Python operator, dispatches to keys.__matmul__) -- NOT elementwise multiply (*).
# keys is (8, 4), query is (4,) -> keys @ query broadcasts to (8,), one score per key,
# equivalent to the whole loop above but run as a single fused BLAS call.

assert torch.allclose(scores_slow, scores_fast)  # `assert cond` -- C++'s assert(), but a plain
                                                  # keyword here, not a macro/header include

weights = F.softmax(scores_fast, dim=0)  # dim=0: softmax along the only axis (8 keys) -> sums to 1
output = weights @ values                # (8,) @ (8, 4) -> (4,), the weighted blend of all values


class SoftLookup(nn.Module):
    # class X(Base): single inheritance, like `class SoftLookup : public nn::Module` in C++,
    # but no header/source split and no explicit access specifier needed.
    def __init__(self, dim):
        super().__init__()  # parent ctor call; C++ would do this in the initialiser list instead
        self.scale = dim ** 0.5  # `**` is exponentiation (C++: std::pow or dim**0.5 doesn't exist,
                                  # you'd write std::sqrt(dim)); self.scale is an instance attribute
                                  # -- no separate header declaration needed, this line both
                                  # declares and initialises it

    def forward(self, query, keys, values):
        scores = (keys @ query) / self.scale
        weights = F.softmax(scores, dim=0)
        return weights @ values

    # NOTE: nn.Module defines __call__ (inherited, not overridden here), and __call__ wraps
    # forward() with hook machinery. Always invoke `lookup(query, keys, values)`, i.e. rely on
    # __call__ -- calling `lookup.forward(...)` directly silently skips every registered hook.


lookup = SoftLookup(dim=d)
print(lookup(query, keys, values))  # invokes __call__ -> forward(), not lookup.forward(...) directly
```

The takeaway that matters architecturally: `keys @ query` followed by `softmax` followed by `weights @ values` is the *entire* forward pass of attention. Every "fancier" attention variant you'll meet later (scaled, multi-head, causal-masked) is this same three-line skeleton with extra bookkeeping wrapped around it — nothing about the core operation changes.

## 📡 SIGNAL LAB

Soft lookup is not a new idea to DSP — it's matched filtering wearing a different name. In matched filtering / radar-and-sonar detection, you cross-correlate a received signal against a bank of known reference templates and the correlation value at each template *is* your similarity score; a hard decision just takes the argmax template ("it's target class 3"). Attention is what happens when instead of hard-deciding, you keep the entire correlation profile and use it as a weighting.

**Problem.** You have a bank of 4 reference "template" signals (each length 16, synthetic) and one received query signal that's a noisy, amplitude-scaled copy of template 2. Compute the normalized cross-correlation (at zero lag, i.e. a plain dot product after unit-normalizing each vector — this keeps the score in $[-1, 1]$ and independent of amplitude, exactly like cosine similarity) between the query and each template, softmax those correlations into weights, and produce the soft-lookup output as a weighted blend of the templates themselves (using each template as its own "value").

```python
import numpy as np
np.random.seed(42)

t = np.linspace(0, 1, 16, endpoint=False)
templates = np.stack([np.sin(2*np.pi*f*t) for f in [3, 7, 11, 15]])  # 4 templates, distinct freqs
query = 0.6 * templates[2] + 0.25 * np.random.randn(16)  # noisy, scaled copy of template 2

# normalized cross-correlation at zero lag == cosine similarity
def unit(v): return v / np.linalg.norm(v)
scores = np.array([unit(query) @ unit(tpl) for tpl in templates])
weights = np.exp(scores) / np.exp(scores).sum()
output = weights @ templates

print("scores :", np.round(scores, 3))   # [-0.048  0.083  0.847 -0.079]
print("weights:", np.round(weights, 3))  # [0.187  0.212  0.514  0.187]
print("hard pick would be template", scores.argmax())  # 2 -- correct, but a single frequency only
```

**So what.** The hard matched-filter decision ("it's template 2") throws away real, if smaller, correlation with templates 0 and 1 — evidence that might matter if the true signal is actually a *mixture* of frequency components rather than one clean template plus noise, which is the realistic case in almost any generative-forensics setting (a GAN- or diffusion-model image residual isn't a pure single-frequency artifact; it's a superposition of spectral signatures). Soft lookup's weighted output is a genuine linear combination of all four templates — 51% template 2, but also meaningfully ~19–21% templates 0, 1, and 3 — which is a strictly more expressive readout of "what's actually going on in this signal" than a one-hot argmax. This is precisely the property that makes attention so effective as a spectral evidence-combiner rather than a hard classifier: it's the differentiable, trainable generalization of matched-filter bank correlation you already know from DSP, and it's worth keeping this analogy loaded when you get to Day 88 (spectral signatures of generated images) and start asking what a learned attention head over frequency-domain features is *actually* computing.

## 🏋️ THE GAUNTLET

**Problem: Weighted Random Pick**

You're given an array of $n$ positive integer weights $w_1, \dots, w_n$. Design a structure supporting:

- **Preprocessing** the array once, in $O(n)$.
- `pickIndex()`: return index $i$ with probability exactly $\dfrac{w_i}{\sum_j w_j}$, called up to $q$ times, each call **must run in $O(\log n)$**, not $O(n)$.

This is the sampling cousin of today's concept: instead of computing a full soft blend over every weight (attention's `output = Σ w_i v_i`), you're drawing a *single* discrete index proportional to that same kind of weight distribution — the same "some items matter more than others" structure, but realized as a sampling problem instead of a weighted-sum problem.

**Constraints:**
- $1 \le n \le 10^4$, $1 \le w_i \le 10^5$
- Up to $q \le 10^4$ calls to `pickIndex()`
- Each call: target $O(\log n)$ time; preprocessing: target $O(n)$ time, $O(n)$ space

**3 hints (escalating):**
1. You can't binary-search over the raw weights array directly — there's no ordering property to exploit. But if you turn the weights into a *running total* (prefix sums), what property does that transformed array have that the original didn't?
2. The prefix-sum array is monotonically non-decreasing. Draw one uniform random integer in $[1, \text{total sum}]$ — that target lands inside exactly one weight's "slice" of the number line. Which standard binary-search primitive finds "the first prefix sum $\ge$ target" in $O(\log n)$?
3. Precompute `prefix[i] = w_1 + ... + w_i` once, in $O(n)$. Each `pickIndex()` call: draw `target` uniformly in `[1, prefix[n-1]]`, then binary-search (`lower_bound` in C++) for the first index where `prefix[index] >= target`; return that index. Correctness: the number line $[1, \text{total}]$ is partitioned into $n$ contiguous slices of length $w_1, w_2, \dots, w_n$ in order, so a uniform draw over the whole line lands in slice $i$ with probability exactly $w_i / \text{total}$ — which is exactly the distribution required.

**Pattern:** prefix sums + binary search for weighted sampling. Target: $O(n)$ preprocessing, $O(\log n)$ per query.

🔓 GAUNTLET SOLUTION

```cpp
#include <bits/stdc++.h>
using namespace std;

// Weighted random pick via prefix sums + binary search. Preprocessing turns the
// weight array into a monotonically non-decreasing prefix-sum array in O(n);
// each pick() draws a uniform target on the number line [1, total] and binary
// searches for the slice it lands in, in O(log n).
class WeightedPicker {
public:
    WeightedPicker(vector<int>& w) : rng(random_device{}()) {
        prefix.resize(w.size());
        long long running = 0;
        for (size_t i = 0; i < w.size(); ++i) {
            running += w[i];
            prefix[i] = running;
        }
        dist = uniform_int_distribution<long long>(1, running);
    }

    int pickIndex() {
        long long target = dist(rng);
        // lower_bound: first element >= target -- the slice this target falls into
        int idx = lower_bound(prefix.begin(), prefix.end(), target) - prefix.begin();
        return idx;
    }

private:
    vector<long long> prefix;
    mt19937_64 rng;
    uniform_int_distribution<long long> dist;
};

int main() {
    int n;
    cin >> n;
    vector<int> w(n);
    for (auto& x : w) cin >> x;

    WeightedPicker picker(w);

    int q;
    cin >> q;
    while (q--) {
        cout << picker.pickIndex() << " ";
    }
    cout << "\n";
    return 0;
}
```

Complexity: preprocessing builds the prefix-sum array in $O(n)$; each `pickIndex()` does one $O(1)$ random draw and one $O(\log n)$ binary search, so $q$ queries cost $O(n + q \log n)$ total, $O(n)$ space.

💡 CONCEPT ANSWER

**Two properties softmax has that naive linear normalization ($w_i = e_i / \sum_j e_j$) doesn't:**

1. **It's always a valid probability distribution, regardless of the sign or scale of the raw scores.** Similarity scores $e_i = q \cdot k_i$ can be negative — dot products of arbitrary real-valued vectors have no sign guarantee. Dividing raw scores by their sum can produce *negative weights* (if some $e_i < 0$) or even divide by a sum that's zero or negative, which isn't a probability distribution at all. Softmax's $\exp(\cdot)$ guarantees every weight is strictly positive before normalizing, so $w_i \in (0, 1)$ and $\sum_i w_i = 1$ always holds, no matter what raw scores come out of the dot products.

2. **It's differentiable everywhere and has well-behaved gradients**, which matters enormously since attention weights need to be learned end-to-end via backprop. $\partial w_i / \partial e_j$ has a clean closed form (the standard softmax Jacobian, $w_i(\delta_{ij} - w_j)$) that's smooth and bounded. A hard argmax-style selection is non-differentiable (zero gradient almost everywhere, undefined at the boundary) — you couldn't backpropagate through "pick the single best key" at all. Softmax is precisely the smooth relaxation that makes hard lookup trainable.

A third property worth knowing for a real interview screen: softmax also *exaggerates* the gap between large and small scores exponentially (a score that's 2 higher than another gets $e^2 \approx 7.4\times$ more weight, not just 2$\times$ more), which is why the **scaled** dot-product in tomorrow's concept — dividing by $\sqrt{d}$ before the softmax — exists: unscaled dot products in high dimensions can grow large enough that softmax saturates into an almost-hard argmax, which reintroduces the vanishing-gradient problem softmax's smoothness was supposed to avoid in the first place.
