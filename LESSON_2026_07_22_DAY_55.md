# Day 55 — Scaled Dot-Product Attention (why √d)

## 🧠 CONCEPT OF THE DAY

**Intuition.** Yesterday you built Q, K, V — a query asking "who's relevant to me?" and keys answering "here's what I contain." The natural way to score relevance is a dot product: `q · k` is large when the two vectors point the same direction, small/negative otherwise. That's a fine similarity score... in low dimensions. But `d_k` isn't small — it's often 64, 128, or more. Sum up 128 roughly-independent terms and the *variance* of that sum explodes, even though each individual term is well-behaved. A handful of scores end up enormous compared to the rest, softmax slams almost all its probability mass onto the biggest one, and every other logit gets a near-zero gradient. The fix isn't architectural — it's a single division. Divide by `√d_k` and you cancel out exactly the variance growth that dimensionality introduced, so attention behaves the same way whether `d_k` is 8 or 512.

**The math.** For a query vector `q ∈ ℝ^d` and key vector `k ∈ ℝ^d` with independent components of mean 0 and variance 1:

$$\text{Var}(q \cdot k) = \text{Var}\left(\sum_{i=1}^{d} q_i k_i\right) = \sum_{i=1}^{d} \text{Var}(q_i k_i) = d$$

(each term `q_i k_i` is a product of two independent unit-variance variables, which itself has variance 1 — sum d of them, variance is d). So the standard deviation of the raw dot product grows as `√d`. Dividing the score by `√d` before the softmax restores unit variance regardless of dimension:

$$\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^{\top}}{\sqrt{d_k}}\right) V$$

where `Q ∈ ℝ^{n×d_k}`, `K ∈ ℝ^{n×d_k}`, `V ∈ ℝ^{n×d_v}`, and the softmax is applied row-wise over the `n` keys each query attends to.

**Why it matters / where it leads.** This one division is what makes attention *trainable* at realistic dimensions — without it, deep transformer stacks would see softmax saturate early in training and gradients would die exactly where you need them most: distinguishing which of several relevant tokens matters most. It sets up Day 56 (multi-head attention), where instead of one big `d_k`-dimensional dot product, you'll split into several smaller heads — each with its own, cheaper, well-scaled `QK^⊤`. A senior interviewer's favorite follow-up to "explain attention": *"Why √d specifically, and not d, or log(d)?"* — if you can derive the variance argument above from scratch on a whiteboard, you've answered it correctly.

**Interview-style question:** If you replaced `√d_k` with `d_k` (i.e., divided by the full dimension instead of its square root), what would happen to the attention distribution as `d_k` grows large — and why is that a *worse* failure mode than not scaling at all?

## 🐍 PYTHONIC EDGE

The bad way: hand-roll every step of scaled dot-product attention with separate ops — readable once, but slow (multiple kernel launches, extra memory traffic for the intermediate score matrix) and easy to get the masking/scaling order wrong. The clean way: PyTorch 2.x ships a fused, hardware-optimized primitive that does the whole thing — including the √d scaling — in one call.

```python
import torch
import torch.nn.functional as F

# ---------- bad way: manual, unfused ----------
def attention_manual(q, k, v, mask=None):
    d_k = q.size(-1)                          # .size(-1) == last-dim length; like .shape[-1]
    # q @ k.transpose(-2, -1): @ is Python's matmul operator (no C++ equivalent operator)
    scores = q @ k.transpose(-2, -1) / (d_k ** 0.5)   # ** is exponentiation, not XOR like C/C++
    if mask is not None:
        # masked_fill returns a NEW tensor (out-of-place); masked_fill_ would mutate in place
        scores = scores.masked_fill(mask == 0, float("-inf"))
    weights = F.softmax(scores, dim=-1)       # dim=-1 = last axis, Python negative indexing
    return weights @ v                        # second @: weighted sum over values

# ---------- clean way: fused kernel (does the √d scaling internally) ----------
class SelfAttention(torch.nn.Module):
    # class X(Base): single-inheritance syntax; C++ would write class X : public Base
    def __init__(self, d_model):
        # def __init__(self, ...): constructor; `self` is explicit (C++'s implicit `this`)
        super().__init__()                    # parent ctor call; C++ uses an initializer list instead
        self.d_model = d_model                # instance attribute — no header declaration needed

    def forward(self, q, k, v, mask=None):
        # F.scaled_dot_product_attention: fused, flash-attention-backed, handles /sqrt(d) for you
        return F.scaled_dot_product_attention(q, k, v, attn_mask=mask)

attn = SelfAttention(d_model=64)
q, k, v = (torch.randn(2, 8, 64) for _ in range(3))   # generator expression, no C++ equivalent
out = attn(q, k, v)   # calling the object invokes __call__, which wraps forward() with hooks —
                       # never call attn.forward(q, k, v) directly, you'd skip those hooks
```

One line does the work of five, skips materializing the full `n×n` score matrix in memory (that's the "flash" part), and is the version you should default to in real code — `attention_manual` above exists only to make the √d step visible.

## 📡 SIGNAL LAB

Attention's `q · k` is a correlation — the same operation you already know from matched filtering. In radar/comms, if you cross-correlate a received signal against a known template of length `N` (coherent integration over `N` samples), the *signal* term in the correlation output grows linearly with `N` while the *noise* term's standard deviation only grows as `√N` — giving you the classic **processing gain of √N** in SNR. That's a *good* thing there: more samples, better detection, and you don't normalize it away because you want the raw correlation peak to stand out.

Attention has the mirror-image problem: `q_i k_i` behaves like *noise-times-noise* (both q and k start as roughly zero-mean, unit-variance projections with no true correlation structure to exploit at initialization), so the "gain" from summing `d_k` such terms is pure variance inflation with no informative signal backing it. Scaling by `1/√d_k` is exactly *undoing* the matched-filter processing gain — because here it's an artifact of dimensionality, not evidence of a match.

**Worked check**, via the convolution theorem (correlation = IFFT of the elementwise conjugate product of FFTs):

```python
import numpy as np
np.random.seed(42)

N = 256
template = np.random.randn(N)                 # our "key"
signal = np.roll(template, 30) + 0.5 * np.random.randn(N)  # shifted template + noise = our "query" context

# cross-correlation via FFT (convolution theorem) instead of O(N^2) direct sliding-dot-product
corr = np.fft.ifft(np.fft.fft(signal) * np.conj(np.fft.fft(template))).real
peak_lag = np.argmax(corr)
peak_val = corr.max()
noise_floor_std = corr.std()

print(f"detected shift: {peak_lag} (true shift: 30)")
print(f"peak / noise-floor ratio: {peak_val / noise_floor_std:.2f}")
```

The peak at lag 30 is real signal — you want its magnitude to grow with `N` (more template samples = sharper, more confident detection), so you leave it unnormalized or normalize by energy for a *bounded correlation coefficient* (NCC), not by `√N` mid-computation. Attention's `q · k`, in contrast, has no true underlying shift to find at initialization — it's pure high-dimensional noise, so the "so what" is: **scaling is a decision about whether growth in your statistic reflects real structure (matched filter: keep it) or dimensionality noise (attention: cancel it).** Same operation, opposite normalization call, entirely dependent on what's actually driving the magnitude.

## 🏋️ THE GAUNTLET

**Problem: Kth Smallest Element in a Sorted Matrix**

You're given an `n × n` matrix where every row and every column is sorted in ascending order (not globally sorted — think of it as `d_k` independent sorted "channels" you have to merge information across, same flavor of problem as merging attention scores across heads later). Find the `k`-th smallest element in the matrix.

**Constraints:** `1 ≤ n ≤ 300`, `1 ≤ k ≤ n²`, matrix values fit in a 32-bit signed int (may include duplicates).

**Target:** `O(n log(max - min))` time, `O(1)` extra space (no heap of size k).

**Hints (use only if stuck, in order):**
1. A max-heap of size `k` or a min-heap merge across rows gets you a correct but `O(k log n)`-ish solution — that's your fallback, not your target. What totally different search space could you binary-search over instead of *positions* in the matrix?
2. You don't need to binary-search over indices — binary-search over the **value range** `[min, max]` of the matrix itself. For a candidate value `mid`, ask: "how many elements in the matrix are ≤ mid?" That count is monotonic in `mid`, which is exactly what binary search needs.
3. Counting "how many elements ≤ mid" naively is `O(n²)`. Because rows *and* columns are sorted, you can do that count in `O(n)` with a staircase walk: start at the bottom-left corner and move up/right depending on whether the current element is ≤ mid or not.

**Pattern:** binary search on the *answer/value space*, not on array indices — recognize this pattern whenever a problem gives you a monotonic "count of elements satisfying X" as a function of a threshold.

🔓 GAUNTLET SOLUTION

```cpp
class Solution {
public:
    int kthSmallest(vector<vector<int>>& matrix, int k) {
        int n = matrix.size();
        int lo = matrix[0][0], hi = matrix[n - 1][n - 1];

        while (lo < hi) {
            int mid = lo + (hi - lo) / 2;
            int count = countLessEqual(matrix, mid, n);

            if (count < k) {
                lo = mid + 1;   // not enough elements <= mid, search higher
            } else {
                hi = mid;       // at least k elements <= mid, answer could be mid or lower
            }
        }
        return lo;
    }

private:
    // staircase walk: start bottom-left, O(n) per call
    int countLessEqual(vector<vector<int>>& matrix, int target, int n) {
        int count = 0;
        int row = n - 1, col = 0;

        while (row >= 0 && col < n) {
            if (matrix[row][col] <= target) {
                // everything above this cell in the current column is also <= target
                count += row + 1;
                col++;
            } else {
                row--;
            }
        }
        return count;
    }
};
```

`countLessEqual` is `O(n)` (row only decreases, col only increases, each moves at most n times), and the binary search runs `O(log(max - min))` times, giving the target `O(n log(max - min))` overall.

💡 CONCEPT ANSWER

Dividing by `d_k` instead of `√d_k` **over-corrects**: since the dot product's standard deviation only grows as `√d_k`, dividing by the full `d_k` shrinks the scaled scores toward zero as `d_k → ∞` (they'd scale as `1/√d_k → 0`). Every score converges to roughly the same tiny value, so softmax outputs the **uniform distribution** regardless of which keys are actually most relevant — the model loses all ability to discriminate. That's arguably worse than no scaling at all: unscaled attention at least concentrates on *something* (even if too sharply and with vanishing gradients on the losers); over-scaled attention collapses to "pay equal attention to everything," discarding the query-key relationship's signal entirely. `√d_k` is the unique exponent that keeps variance constant — anything less under-corrects toward saturation, anything more over-corrects toward uniformity.
