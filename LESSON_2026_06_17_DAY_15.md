## 🧠 CONCEPT OF THE DAY

### Intuition: Let Noisy Parameters Rest

After 14 days you know momentum and Nesterov help you move faster in consistent directions. But what if different parameters have *wildly* different gradient magnitudes? A weight in a rare word embedding might update once per thousand steps; a shared bias updates every step. A single global learning rate η treats both the same — that's unfair and inefficient.

**AdaGrad** (Adaptive Gradient, Duchi et al. 2011) fixes this by giving each parameter its own effective learning rate that shrinks proportionally to how often it has been updated. Parameters that receive large, frequent gradients get a *smaller* step; rare-but-large gradient parameters get a *larger* relative step.

**RMSprop** (Hinton, unpublished 2012) patches AdaGrad's fatal flaw: the accumulator only grows, so learning eventually stalls. RMSprop replaces the sum with an exponential moving average (EMA), giving the optimizer a fading memory.

---

### The Math

**AdaGrad:**

Let $g_t = \nabla_\theta \mathcal{L}_t$ be the gradient at step $t$, one scalar per parameter.

Accumulate squared gradients:

$$G_t = G_{t-1} + g_t^2, \quad G_0 = 0$$

Update rule:

$$\theta_{t+1} = \theta_t - \frac{\eta}{\sqrt{G_t + \varepsilon}} \cdot g_t$$

- $G_t$ — cumulative sum of squared gradients (per-parameter scalar)
- $\varepsilon \approx 10^{-8}$ — numerical stabilizer so we never divide by zero
- Effective LR: $\eta_t = \eta / \sqrt{G_t}$ — monotonically decreasing. Good early on; lethal long-term.

**RMSprop:**

Replace the sum with a decaying average:

$$v_t = \rho \, v_{t-1} + (1 - \rho) \, g_t^2, \quad v_0 = 0$$

$$\theta_{t+1} = \theta_t - \frac{\eta}{\sqrt{v_t + \varepsilon}} \cdot g_t$$

- $\rho \in [0.9, 0.99]$ — decay factor (typical: 0.9)
- $v_t$ — exponential moving average of $g^2$ (tracks recent gradient energy, not all-time sum)
- Effective learning rate stays bounded even after many steps

**Why $\varepsilon$ matters:** without it, a parameter that receives zero gradient forever has $G_t = 0$ and causes division by zero on the first non-zero gradient.

---

### Why it matters / where it leads

AdaGrad's insight — *per-parameter learning rates* — is the seed for every modern optimizer. RMSprop's EMA of squared gradients is the denominator half of **Adam** (Day 16). The moment you see $v_t$ in Adam, remember it came directly from here.

Interview angle: "How does AdaGrad handle sparse gradients?" A sparse embedding column gets tiny $G_t$ (rarely updated), so its effective LR stays large — exactly what you want. Transformers use Adam precisely because sparse token embeddings benefit from this adaptive scaling.

---

**Interview question (answer at bottom):**
> "AdaGrad is great for sparse features but fails for deep networks trained over many steps. Why? And what single change does RMSprop make to fix it?"

## 🐍 PYTHONIC EDGE

### In-place EMA updates — avoid the slow loop

When implementing a custom optimizer you often need to update a running second-moment estimate for each parameter. The **bad way** uses Python-level operations or `**` inside a grad loop:

```python
# BAD — Python loop, creates new tensors, autograd overhead
for p in model.parameters():
    v[p] = rho * v[p] + (1 - rho) * (p.grad ** 2)
    p.data -= lr / (v[p].sqrt() + eps) * p.grad
```

The **clean way** uses `torch.Tensor.mul_` + `torch.Tensor.addcmul_` (in-place, fused, no temp allocation) and accesses `.grad.data` to skip autograd tracking:

```python
# GOOD — in-place, vectorized, no autograd overhead
for p in model.parameters():
    g = p.grad.data                      # skip autograd wrapper
    v[p].mul_(rho).addcmul_(g, g, value=1 - rho)   # v = rho*v + (1-rho)*g²
    p.data.addcdiv_(g, v[p].sqrt().add_(eps), value=-lr)  # θ -= lr*g/√(v+ε)
```

`addcmul_(tensor1, tensor2, value=scalar)` computes `self += scalar * tensor1 * tensor2` — that's the squared gradient EMA in one kernel call. `addcdiv_` does the parameter update in one more. PyTorch's built-in `torch.optim.RMSprop` does exactly this under the hood.

## 📡 SIGNAL LAB

### RMSprop as a first-order IIR low-pass filter on gradient power

Your research lane: frequency-domain analysis. Here's how RMSprop maps directly onto classical DSP.

The RMSprop update on the squared gradient is:

$$v_t = \rho \, v_{t-1} + (1 - \rho) \, g_t^2$$

This is a **first-order recursive (IIR) filter** applied to the signal $s_t = g_t^2$. In z-domain notation:

$$V(z) = \frac{(1 - \rho)}{1 - \rho z^{-1}} \cdot S(z)$$

The pole is at $z = \rho$. Since $\rho \in (0, 1)$, the pole is strictly inside the unit circle → BIBO stable. The frequency response magnitude is:

$$|H(e^{j\omega})| = \frac{1-\rho}{\sqrt{1 + \rho^2 - 2\rho\cos\omega}}$$

This is a **low-pass filter**: DC gain is 1, high-frequency gain → $(1-\rho)/(1+\rho)$.

**What does "low-pass on $g^2$" mean for training?**

- High-frequency spikes in $g_t$ (noisy minibatch directions) get attenuated in $v_t$.
- The effective LR $\eta / \sqrt{v_t}$ is therefore *stabilized* against gradient noise bursts.
- The filter time constant is $\tau = 1/(1-\rho)$ steps. At $\rho = 0.9$: $\tau = 10$ steps — the optimizer "remembers" the last ~10 gradient magnitudes. At $\rho = 0.99$: $\tau = 100$ steps — much longer memory.

**Signal Lab experiment:** Generate a gradient signal $g_t = \sin(0.05t) + 0.5 \cdot \xi_t$ where $\xi_t \sim \mathcal{N}(0,1)$ and plot $v_t$ for $\rho \in \{0.5, 0.9, 0.99\}$. You'll see the EMA act exactly like a leaky integrator — higher $\rho$ gives a smoother but slower-to-respond estimate, lower $\rho$ tracks noise.

**Forensics tie-in:** In your frequency-domain generative forensics work, spectral artifacts from upsampling (periodic spikes in the gradient spectrum during GAN/diffusion training) can cause Adam/RMSprop to over-shrink learning rates at specific frequencies. This is one reason why Adam can sometimes under-fit high-frequency texture components compared to SGD — the $v_t$ denominator grows large at artifact frequencies and suppresses updates there.

## 🏋️ THE GAUNTLET

### Problem: Minimum Sum-of-Squares Window

You are given an integer array `arr` of length `n` and an integer `k`. Find the **starting index** of the contiguous subarray of length exactly `k` that has the minimum sum of squared elements. If multiple windows tie, return the smallest (leftmost) starting index.

**Input format:**
- First line: `n k`
- Second line: `n` integers

**Constraints:**
- $1 \le k \le n \le 10^5$
- $|arr_i| \le 10^4$

**Example:**
```
5 3
1 -3 2 1 -1
```
Output: `1 6`  (starting index=1, sum-of-squares=9+4+1=14 ... wait)

Actually: index 0: 1+9+4=14 | index 1: 9+4+1=14 | index 2: 4+1+1=6 ← minimum
Output: `6 2`

---

**Hint 1:** Can you precompute prefix sums of squares in O(n) so that any window query is O(1)?

**Hint 2:** `prefix[i+k] - prefix[i]` gives the sum of squares in the window `[i, i+k-1]`. How do you build `prefix`?

**Hint 3:** After building the prefix array, a single left-to-right scan finds the minimum. No sorting needed.

---

**Pattern:** Prefix sums on squared values → O(n) precompute, O(1) per query, O(n) total.

**Target complexity:** O(n) time, O(n) space (reducible to O(1) extra with a sliding variable).

**Why this connects to today's concept:** AdaGrad's accumulator $G_t = \sum_{\tau=1}^{t} g_\tau^2$ is a prefix sum of squared gradients — and computing window statistics on squared signals is the same primitive operation that underlies both the algorithm and this problem.

## 🔓 GAUNTLET SOLUTION

```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, k;
    cin >> n >> k;
    vector<long long> arr(n);
    for (auto& x : arr) cin >> x;

    // Prefix sum of squares: prefix[i+1] = prefix[i] + arr[i]^2
    vector<long long> prefix(n + 1, 0LL);
    for (int i = 0; i < n; i++)
        prefix[i + 1] = prefix[i] + arr[i] * arr[i];

    long long best = LLONG_MAX;
    int best_idx = 0;

    for (int i = 0; i + k <= n; i++) {
        long long window_ss = prefix[i + k] - prefix[i];
        if (window_ss < best) {
            best = window_ss;
            best_idx = i;
        }
    }

    cout << best << " " << best_idx << "\n";
    return 0;
}
```

**Why it works:** `prefix[i+k] - prefix[i]` = $\sum_{j=i}^{i+k-1} arr_j^2$ in O(1) after an O(n) build. One scan finds the minimum. Total: O(n).

**Sliding variable alternative (O(1) space):** Maintain a running `window_ss` variable; on each step subtract `arr[i-1]^2` and add `arr[i+k-1]^2`. Same O(n) time, only O(1) extra space beyond the input.

## 💡 CONCEPT ANSWER

**AdaGrad's problem:** the accumulator $G_t = \sum_{\tau=1}^{t} g_\tau^2$ grows **monotonically and without bound**. The effective learning rate $\eta / \sqrt{G_t}$ therefore shrinks toward zero and never recovers — training stalls, often long before convergence in deep networks trained for many epochs.

**RMSprop's fix:** replace the cumulative sum with an **exponential moving average** $v_t = \rho v_{t-1} + (1-\rho)g_t^2$. This gives the optimizer a finite memory of length $\approx 1/(1-\rho)$ steps. Recent gradient magnitudes dominate; old gradients fade away. The effective LR stays bounded and can recover if gradient magnitudes decrease — crucial for deep networks with non-stationary gradient statistics.
