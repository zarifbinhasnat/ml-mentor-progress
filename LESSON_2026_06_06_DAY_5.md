## 🧠 CONCEPT OF THE DAY

### Intuition
Softmax is how a network turns a pile of raw, unbounded scores ("logits") into something that behaves like a probability distribution: every value lands in $(0,1)$ and they all sum to 1. Cross-entropy is then how you score that distribution against the truth — it asks "how surprised should I be by the correct answer, given the probability you assigned it?" Confidently right costs you almost nothing; confidently *wrong* costs you enormously. That asymmetric, logarithmic punishment is the entire engine of classification training.

### The Math

**Softmax** (converts logits $\mathbf{z} \in \mathbb{R}^C$ into a distribution):

$$\text{softmax}(\mathbf{z})_j = \frac{e^{z_j}}{\sum_{c=1}^{C} e^{z_c}}$$

**Cross-entropy loss** (for one-hot target with true class $y$):

$$\mathcal{L}_{\text{CE}} = -\log\big(\text{softmax}(\mathbf{z})_y\big) = -z_y + \log\sum_{c=1}^{C} e^{z_c}$$

**The gradient — the single most beautiful cancellation in this whole curriculum:**

$$\frac{\partial \mathcal{L}_{\text{CE}}}{\partial z_j} = \text{softmax}(\mathbf{z})_j - \mathbb{1}[j = y] = \hat{p}_j - p_j$$

| Symbol | Meaning |
|--------|---------|
| $\mathbf{z} \in \mathbb{R}^C$ | raw logits over $C$ classes |
| $\hat{p}_j = \text{softmax}(\mathbf{z})_j$ | predicted probability of class $j$ |
| $p_j = \mathbb{1}[j=y]$ | true (one-hot) probability of class $j$ |
| $y$ | index of the correct class |

Read that gradient again: **predicted minus true probability.** No sigmoid-derivative term, no chain-rule mess — softmax's own derivative cancels perfectly against cross-entropy's logarithm. This is the *exact same cancellation* that resolved yesterday's MSE-on-sigmoid pathology, generalized from binary to multi-class.

### Why it matters / where it leads
Softmax + cross-entropy isn't a folk-wisdom pairing — it *is* the maximum-likelihood loss for a categorical (multinoulli) distribution, the natural extension of the Gaussian/MSE story from yesterday. That clean $\hat p - p$ gradient is why this pairing is fused into a single numerically-stable operation (`CrossEntropyLoss` in PyTorch, taking raw logits) in virtually every classifier ever trained — from MNIST logistic regression to GPT's next-token prediction head (Concept 68 makes this connection explicit: language modeling *is* per-token classification over the vocabulary). Whenever you see "the model outputs logits, then we apply softmax and cross-entropy," you're watching the same mathematical machine that powers everything from spam filters to trillion-parameter LLMs.

## 🐍 PYTHONIC EDGE

**Trick:** Never manually chain `softmax` then `log` then `nll_loss` — use the fused, log-sum-exp-stabilized `cross_entropy`, which sidesteps overflow from `exp` of large logits entirely.

```python
import torch
import torch.nn.functional as F

logits = torch.tensor([[1000.0, 1001.0, 999.0]])   # deliberately huge logits
target = torch.tensor([1])

# BAD — naive chain: exp(1000) overflows to inf, softmax becomes nan
def ce_bad(logits, target):
    probs = torch.exp(logits) / torch.exp(logits).sum(dim=-1, keepdim=True)  # nan
    return -torch.log(probs[range(len(target)), target]).mean()              # nan

# GOOD — fused, log-sum-exp trick subtracts the max internally before exponentiating
loss = F.cross_entropy(logits, target)   # finite, correct: ≈ 0.4076

# If you need probabilities for inspection (not for the loss!), use log_softmax:
log_probs = F.log_softmax(logits, dim=-1)   # stable; exponentiate only for display
```

**Bonus gotcha:** `nn.CrossEntropyLoss` *internally* applies `log_softmax` — feeding it already-softmaxed probabilities silently double-squashes your distribution and trains a subtly broken model that often "sort of works," which makes the bug brutal to catch in review.

## 📡 SIGNAL LAB

### Cross-Entropy as a Spectral Similarity Score

A power spectral density (PSD) is itself a probability-like distribution once normalized — it's nonnegative and can be made to sum to 1. That means cross-entropy (and its symmetric cousin, KL divergence) can directly compare *where two signals concentrate their energy across frequency*, not just their raw amplitudes.

**Problem:** Generate two signals: $x_1[n] = \sin(2\pi \cdot 0.1 n) + 0.5\sin(2\pi \cdot 0.3 n)$ and $x_2[n] = \sin(2\pi \cdot 0.1 n) + 0.5\sin(2\pi \cdot 0.32 n)$ (a slightly shifted second harmonic), $n = 0,\dots,511$. Normalize their power spectra into probability distributions and compute the cross-entropy between them. What does a *low* vs *high* value tell you physically?

**Worked solution:**

```python
import numpy as np

n = np.arange(512)
x1 = np.sin(2*np.pi*0.10*n) + 0.5*np.sin(2*np.pi*0.30*n)
x2 = np.sin(2*np.pi*0.10*n) + 0.5*np.sin(2*np.pi*0.32*n)

def psd_dist(x):
    P = np.abs(np.fft.rfft(x))**2
    return P / P.sum()           # now a valid probability distribution over frequency bins

p, q = psd_dist(x1), psd_dist(x2)
eps = 1e-12
cross_entropy = -np.sum(p * np.log(q + eps))
self_entropy = -np.sum(p * np.log(p + eps))
# cross_entropy > self_entropy whenever p != q — the gap IS the KL divergence,
# and it grows as the harmonic shift (0.30 → 0.32) grows
```

**So what:** The gap $H(p,q) - H(p) = D_{KL}(p \| q)$ quantifies *exactly* how much "extra surprise" you incur by describing signal $x_1$'s spectrum using $x_2$'s spectrum as your model. This is not an analogy — spectral cross-entropy and KL divergence are used verbatim in audio/speech model evaluation and in comparing real vs. generated image spectra (Concept 88). Internalizing cross-entropy as "a distance between distributions" rather than "the classification loss" is what lets you carry this tool into completely different domains — including your own frequency-domain forensics work, where comparing real vs. synthetic spectral *distributions* is a first-class technique.

## ⚔️ THE GAUNTLET

### Maximum Likelihood Partition

You're given `n` events, each with an observed outcome `obs[i]` (an integer class label in `[0, C)`) and a model that assigns probability `prob[i][c]` to class `c` for event `i`. You must select a contiguous subarray of events of length **exactly** `k` that **minimizes total cross-entropy loss** $-\sum_{i \in S} \log(\text{prob}[i][\text{obs}[i]])$.

**Constraints:**
- $1 \leq k \leq n \leq 10^5$
- $1 \leq C \leq 50$
- $10^{-6} \leq \text{prob}[i][c] \leq 1$ (rows sum to 1)
- Time: $O(n)$ after $O(nC)$ preprocessing, Space: $O(n)$

**Input format:**
```
5 2 3
0 0.7 0.2 0.1
1 0.1 0.8 0.1
2 0.3 0.3 0.4
0 0.6 0.3 0.1
1 0.2 0.7 0.1
```
(each event line: `obs[i]` followed by `prob[i][0..C-1]`)

**Hints:**
1. First collapse each event to a single number: $\ell_i = -\log(\text{prob}[i][\text{obs}[i]])$. Now the problem is "minimum-sum contiguous subarray of fixed length k" — a much more familiar shape.
2. A brute-force re-sum for every window is $O(nk)$ — too slow. What lets you go from one window's sum to the next window's sum in $O(1)$?
3. Sliding window with a running sum: subtract the element leaving, add the element entering. Track the minimum over all valid windows.

**Pattern:** Precompute + Fixed-Size Sliding Window
**Target:** $O(n)$ time (after $O(nC)$ to build $\ell$), $O(n)$ space

## 🏗️ BLUEPRINT

**System design nugget — Loss computation at vocabulary scale:**
For an LLM with a 100K-token vocabulary, the final layer's logits tensor for a batch of `(B, T)` tokens is `(B, T, 100000)` — often the single largest tensor in the whole forward pass, and computing softmax + cross-entropy over it naively can dominate both memory and runtime. Production training stacks fuse the final linear projection, log-softmax, and NLL loss into a single **chunked/fused cross-entropy kernel** (e.g., Liger, `torch.compile`-fused CE) that never materializes the full `(B, T, V)` probability tensor in memory — computing loss (and its gradient) chunk-by-chunk over the vocabulary dimension. The tradeoff is implementation complexity vs. a frequently 2-4× reduction in peak memory at the loss computation step, often the difference between fitting a larger batch or not.

## 🔒 GAUNTLET SOLUTION

```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    ios_base::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, k, C;
    cin >> n >> k >> C;
    vector<double> loss(n);
    for (int i = 0; i < n; i++) {
        int obs;
        cin >> obs;
        vector<double> p(C);
        for (auto& x : p) cin >> x;
        loss[i] = -log(max(p[obs], 1e-12));
    }

    double windowSum = 0.0;
    for (int i = 0; i < k; i++) windowSum += loss[i];
    double best = windowSum;

    for (int i = k; i < n; i++) {
        windowSum += loss[i] - loss[i - k];
        best = min(best, windowSum);
    }
    printf("%.6f\n", best);
    return 0;
}
```

**Why it works:** Reducing each event to its negative log-likelihood collapses a multi-class problem into a 1D array; the task becomes "minimum sum over any contiguous window of fixed length k," solvable by a textbook sliding window that updates the running sum in O(1) per step (add the incoming element, remove the outgoing one). Total work is O(nC) to build the per-event losses plus O(n) to slide.

**Edge cases:** `k == n` means there's only one valid window — the whole array — and the loop never executes the sliding step.

## 🔑 CONCEPT ANSWER

**Question:** "Why does PyTorch's `nn.CrossEntropyLoss` expect raw logits rather than post-softmax probabilities, and why is computing softmax and cross-entropy as two separate steps numerically dangerous at scale?"

**Answer:** Cross-entropy on raw logits is computed as $\mathcal{L} = -z_y + \log\sum_c e^{z_c}$, and that $\log\sum e^{z_c}$ term (the "log-sum-exp") is evaluated using the **shift-by-max trick**: $\log\sum_c e^{z_c} = m + \log\sum_c e^{z_c - m}$ where $m = \max_c z_c$. This keeps every exponent $\leq 0$, so `exp` never overflows, regardless of how large the logits are. If you instead compute softmax first (producing probabilities, which can legitimately underflow to exactly `0.0` for very confident wrong predictions) and *then* take `log`, you get `log(0) = -inf`, and the loss — and every gradient downstream — becomes `nan`. Fusing softmax and cross-entropy into one operation lets the implementation apply the shift-by-max trick *before* any exponentiation happens, which is only possible if it has access to the raw logits. This is precisely why feeding probabilities to `CrossEntropyLoss` is both semantically wrong (double-softmaxing) and throws away the numerical safety net the fused op exists to provide.

### The Confusion Matrix

For a binary classifier, every prediction falls into one of four buckets:

|  | Predicted Positive | Predicted Negative |
|---|---|---|
| **Actually Positive** | TP (True Positive) | FN (False Negative / Type II Error) |
| **Actually Negative** | FP (False Positive / Type I Error) | TN (True Negative) |

**Worked example from the lecture:** 100 patients, 10 have disease. Model flags 8 sick correctly (TP=8), misses 2 (FN=2), wrongly flags 15 healthy (FP=15), clears 75 correctly (TN=75).

### Derived Metrics

**Accuracy** — fraction of all predictions correct:

$$\text{Accuracy} = \frac{TP + TN}{TP + TN + FP + FN}$$

In the example: $(8+75)/100 = 83\%$. **The horror story:** a model that predicts "always healthy" gets $(0+90)/100 = 90\%$ accuracy while catching zero cancer patients. Accuracy is meaningless on imbalanced datasets.

**Precision** — of all positive predictions, how many are actually positive:

$$\text{Precision} = \frac{TP}{TP + FP}$$

Example: $8/(8+15) = 34.8\%$. Precision answers "when you ring the alarm, how often is it real?" Optimizing precision alone → predict positive only when very certain → miss many real positives.

**Recall (Sensitivity / True Positive Rate)** — of all actual positives, how many did you catch:

$$\text{Recall} = \frac{TP}{TP + FN}$$

Example: $8/(8+2) = 80\%$. Recall answers "out of all real cancer patients, how many did you catch?" Optimizing recall alone → predict positive for everyone → precision collapses.

**F1-Score** — harmonic mean of precision and recall:

$$F_1 = 2 \cdot \frac{\text{Precision} \times \text{Recall}}{\text{Precision} + \text{Recall}}$$

The harmonic mean punishes extreme imbalance between P and R — a model with 100% recall and 1% precision gets $F_1 = 1.98\%$, not 50.5%. This is the standard single-number metric when classes are imbalanced.

**False Positive Rate (FPR):**

$$\text{FPR} = \frac{FP}{FP + TN}$$

Example: $15/90 = 16.7\%$. Used as the x-axis of the ROC curve.

### ROC Curve and AUC

Every classifier has a **decision threshold** $\tau$ — predict positive if $\hat{p} > \tau$. As you sweep $\tau$ from 1 down to 0:
- Recall (TPR) rises (you catch more positives)
- FPR also rises (you also flag more negatives)

Plotting **TPR vs FPR** for all thresholds gives the **ROC curve**.

- **Random classifier**: diagonal line (TPR = FPR at every threshold)
- **Perfect classifier**: corner point at (FPR=0, TPR=1)
- **Real models**: curve above the diagonal

**AUC (Area Under the ROC Curve):**

$$\text{AUC} = P(\hat{p}_{pos} > \hat{p}_{neg})$$

AUC equals the probability that a randomly chosen positive example gets a higher predicted score than a randomly chosen negative. AUC = 0.5 → random; AUC = 1.0 → perfect; AUC < 0.5 → consistently wrong (flip predictions).

AUC is **threshold-independent** and **scale-invariant** — it measures ranking quality, not calibration. Two models can have the same AUC but very different precision/recall tradeoffs at a specific threshold.

### Class Imbalance

When one class dominates (e.g., 95% negatives), a model learns to predict "always negative" and achieves high accuracy for free, as in the lecture's cancer example.

**Diagnosis:** accuracy is high but recall on the minority class is near zero. Check F1 or **Balanced Accuracy**:

$$\text{Balanced Accuracy} = \frac{1}{2}\left(\frac{TP}{TP+FN} + \frac{TN}{TN+FP}\right) = \frac{\text{TPR} + \text{TNR}}{2}$$

**Remedies:**

| Method | How it works | When to use |
|---|---|---|
| Class weights in loss | Multiply minority class loss by $N_{maj}/N_{min}$ | Always try first — zero cost |
| Oversample minority (SMOTE) | Synthesize new minority samples via interpolation | Small minority set |
| Undersample majority | Randomly drop majority samples | Very large datasets |
| Threshold tuning | Move $\tau$ away from 0.5 | After training, during deployment |

In PyTorch: `nn.CrossEntropyLoss(weight=torch.tensor([1.0, 9.0]))` up-weights the rare class by 9× instantly, no resampling needed.

### The Connection Back to Cross-Entropy

Cross-entropy is the *training* loss. Precision, recall, F1, and AUC are *evaluation* metrics. You optimize cross-entropy during training because it's differentiable; you report F1/AUC because they directly measure what you care about (did you find the cancer?). They are not the same thing — a model with lower cross-entropy loss can have worse F1 than a threshold-tuned competitor.
