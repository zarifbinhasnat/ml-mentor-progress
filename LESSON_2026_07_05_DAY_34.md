## 🧠 CONCEPT OF THE DAY

### Mental model

Yesterday you saw the convolution operation as a stencil sliding over an image, computing a local weighted sum at every position. Today is about the three knobs that decide *exactly how* that stencil moves and *what happens at the edges*:

- **Kernel (k)** — the size of the stencil itself. Bigger kernel = each output pixel "sees" a larger patch of input in one hop.
- **Stride (s)** — how many pixels you slide the stencil between outputs. Stride 1 = compute at every position (dense). Stride 2 = skip every other position (the feature map shrinks by ~2×ish, decimation).
- **Padding (p)** — how you handle the border, where the stencil would otherwise hang off the edge. Pad with zeros (or reflect/circular) so edge pixels get a valid output too, instead of just losing them.

### The math

For a single spatial dimension, output size is:

$$\text{out} = \left\lfloor \frac{\text{in} + 2p - k}{s} \right\rfloor + 1$$

where `in` is the input size along that axis, `p` is padding per side, `k` is kernel size, `s` is stride.

Two conventions you'll see everywhere in frameworks:

- **"valid" padding**: $p = 0$. Output strictly shrinks every layer.
- **"same" padding** (stride 1 only): choose $p = \frac{k-1}{2}$ (for odd $k$) so `out == in`. The framework does the arithmetic; you just ask for `padding="same"`.

### Why it matters / where it leads

These three numbers are the entire design vocabulary of a CNN's spatial path. They determine:

- **Receptive field growth** (Concept 36) — how much of the original image a deep neuron "sees."
- **Feature map resolution and compute/memory cost** — stride-2 layers are how you cheaply shrink spatial size while growing channel depth.
- **The pooling vs. strided-conv debate** (Concept 35) — pooling downsamples with a *fixed* rule (max/avg); a stride-2 conv downsamples with a *learned* rule. Modern architectures increasingly prefer the latter.
- **A subtle trap**: stride is literally decimation — throwing away samples. If the kernel hasn't first suppressed high-frequency content, decimation aliases, folding spurious frequencies back into your feature map. That's today's Signal Lab, and it's exactly the kind of artifact that shows up as spectral signatures in generative-model outputs.

**Interview-style question:** *You have a 32×32×3 input. You apply a 5×5 convolution with stride 2 and padding 2. What's the output spatial size, and why might a practitioner choose a stride-2 conv here instead of a stride-1 conv followed by a separate max-pool?*

## 🐍 PYTHONIC EDGE

`padding_mode` is an argument people set once and forget — but the choice quietly decides what "edge" even means, and it's the same choice you make in DSP when you pick how to handle boundaries before an FFT.

```python
import torch
import torch.nn as nn

class EdgeAwareConv(nn.Module):                          # single inheritance: like "class X : public Base" in C++
    def __init__(self, channels, k=3, mode="zeros"):      # constructor; self is C++'s "this", but explicit here
        super().__init__()                                # parent ctor call; C++ instead uses an initializer list
        self.conv = nn.Conv2d(                             # self.x = ... is an instance attr (C++: declare in header, set in body)
            channels, channels, k, padding=k // 2, padding_mode=mode  # // is integer division (C++'s int/int truncation)
        )

    def forward(self, x):                                 # never call this directly...
        return self.conv(x)                                # ...module(x) invokes __call__, which wraps forward() with hooks

x = torch.randn(1, 3, 8, 8)                    # NCHW tensor: batch=1, channels=3, H=W=8

bad  = EdgeAwareConv(3, mode="zeros")(x)        # zero pad: edges see a fake "black border" that isn't in the image
good = EdgeAwareConv(3, mode="circular")(x)     # circular pad: wraps around — matches the DFT's implicit periodicity

print(bad.shape == good.shape)                  # True: identical shapes, opposite edge semantics — a silent footgun
```

The bad way isn't wrong, it's *invisible* — `zeros` is the default, so most code pads with an artificial discontinuity at every border without anyone deciding to. If your downstream analysis assumes periodicity (any FFT-based operation does), `padding_mode="circular"` is the one that's actually consistent with the math.

## 📡 SIGNAL LAB

**Claim:** a strided convolution is "filter, then decimate" — and decimation without adequate filtering aliases.

**Setup.** Take a 1D signal with two tones: $f=3$ and $f=20$ cycles over $N=64$ samples. Nyquist for the *original* sampling is 32 cycles, so both tones are legally represented. Now apply a stride-2 "conv" — first convolve with a 3-tap kernel, then keep every other sample.

Two kernels, same stride:

- **All-pass** $[0, 1, 0]$ — the identity. This is what a stride-2 conv looks like if its learned weights happen to pass everything through (no filtering).
- **Lowpass** $[1, 2, 1]/4$ — a binomial blur, i.e. a kernel that actually attenuates high frequencies before you throw samples away.

**What happens** (see the graph): after stride-2, the new Nyquist is 16 cycles/signal. The $f=20$ tone is above that — it has nowhere legal to go, so it **folds back** to $|20 - 32| = 12$ cycles in *both* cases. But the amplitude at that aliased bin tells the real story: with the all-pass kernel it lands at ≈12.9 (barely attenuated from the original ≈25.6), while with the lowpass kernel it lands at ≈4.0 — the blur did its job and knocked most of it down *before* the fold-back could happen.

![Strided convolution as filter-then-decimate: aliasing appears with an all-pass kernel, is suppressed with a lowpass kernel](graph_DAY_34.png)

**So what:** a stride-2 conv layer is only "safe" from aliasing to the extent that its *learned* kernel happens to behave like a lowpass filter in the frequencies it's discarding — nothing forces that during training. This is precisely the mechanism behind "Making Convolutional Networks Shift-Invariant Again" (blur-pool / anti-aliased downsampling): insert a small fixed blur before every stride, cheaply, and shift-invariance improves measurably. And it cuts both ways for your own research lane — the *upsampling* side of this same coin (transposed convs / nearest-neighbor + conv, Concept 43) is exactly why generated images carry checkerboard and periodic spectral artifacts that forensic classifiers pick up on. Same aliasing math, opposite direction.

## 🏋️ THE GAUNTLET

**Sliding Window Maximum**

Given an integer array `nums` and a window size `k`, return an array of the maximum value in each contiguous window of size `k` as it slides from the left end of the array to the right, one step (stride 1) at a time.

**Constraints:** `1 <= k <= nums.length <= 10^5`, `-10^4 <= nums[i] <= 10^4`.

Example: `nums = [1,3,-1,-3,5,3,6,7]`, `k = 3` → `[3,3,5,5,6,7]`.

**Hint 1:** The brute-force scan-every-window approach recomputes the max from scratch each time — that's $O(nk)$. Between window $i$ and window $i+1$, only one element leaves (the left end) and one enters (the right end). What's the minimum information you need to carry forward instead of recomputing?

**Hint 2:** You don't need to know *every* element's rank in the window — only whether an element could *ever* become the max while it's still in range. If a new element is bigger than something already in your working set, does that smaller element have any future use?

**Hint 3:** Maintain indices in a double-ended queue, keeping the values they point to in strictly decreasing order. Before pushing a new index, pop from the back while the back's value is smaller than the incoming value. Pop from the front whenever the front index falls outside the current window.

**Pattern:** Monotonic deque / sliding window. **Target:** $O(n)$ time, $O(k)$ space.

## 🏗️ BLUEPRINT

**BlurPool / anti-aliased downsampling — the tradeoff hiding in every stride-2 layer**

Inserting a small fixed blur kernel immediately before every strided conv or pool (à la "Making Convolutional Networks Shift-Invariant Again") measurably improves shift-invariance and robustness to input jitter — a 1-pixel crop shift shouldn't flip your top-1 prediction, but with naive stride it sometimes does, because of exactly the aliasing you just saw. The cost is a handful of extra FLOPs per downsampling stage and one more hyperparameter (which blur kernel). Worth it when robustness to small input perturbations matters more than shaving the last millisecond of latency (server-side classifiers, anything feeding a downstream forensic/consistency check); skip it for a mobile real-time model where every layer is FLOP-budgeted.

## 🔓 GAUNTLET SOLUTION

```cpp
#include <vector>
#include <deque>
using namespace std;

class Solution {
public:
    vector<int> maxSlidingWindow(vector<int>& nums, int k) {
        deque<int> dq;              // stores indices; values at those indices are strictly decreasing
        vector<int> result;
        result.reserve(nums.size() - k + 1);

        for (int i = 0; i < (int)nums.size(); ++i) {
            // drop indices that fell out of the window on the left
            while (!dq.empty() && dq.front() <= i - k) {
                dq.pop_front();
            }
            // any queued index whose value is smaller than nums[i] can never
            // be the max again while nums[i] is still in range -> discard it
            while (!dq.empty() && nums[dq.back()] < nums[i]) {
                dq.pop_back();
            }
            dq.push_back(i);

            // front of the deque is always the max of the current window
            if (i >= k - 1) {
                result.push_back(nums[dq.front()]);
            }
        }
        return result;
    }
};
```

Each index enters and leaves the deque at most once, so total work is $O(n)$ despite the nested-looking while loops.

💡 CONCEPT ANSWER

Output size: $\lfloor (32 + 2\cdot2 - 5)/2 \rfloor + 1 = \lfloor 31/2 \rfloor + 1 = 15 + 1 = 16$, so the output is $16 \times 16 \times C_{out}$.

Why stride-2 conv over stride-1 conv + max-pool: folding downsampling directly into the conv lets the network *learn* how to combine and discard spatial information (rather than being locked into a fixed max/avg rule), and it's cheaper — one pass over the data instead of a full-resolution conv followed by a separate pooling pass. The tradeoff (see today's Signal Lab and Blueprint) is that a learned stride-2 kernel isn't guaranteed to behave like an anti-aliasing filter before it decimates, whereas an explicit blur+pool pipeline is.
