## 🧠 CONCEPT OF THE DAY

### Mental model

You already have every ingredient on the table — convolution (Concept 32), stride/padding (33), feature maps (34), pooling (35), receptive fields (36), and 1×1 channel-mixing (37). LeNet and AlexNet aren't new math; they're the two moments in history where someone stacked those ingredients into a working recipe and proved it at scale. Think of them as a before/after photo of the same idea, fourteen years apart.

**LeNet-5 (1998, Yann LeCun)**: a small, elegant stack — `conv → pool → conv → pool → conv → fc → fc` — built to read handwritten zip-code digits for the US Postal Service. Five learnable layers, 32×32 grayscale input, 10 classes, ~60K parameters. It worked, it was published, and then... the field mostly ignored it for over a decade. Why? Not enough labeled data, not enough compute, and the wrong nonlinearity (tanh/sigmoid) for going any deeper — you'll see exactly why in the right panel below.

**AlexNet (2012, Krizhevsky/Sutskever/Hinton)**: the *same architectural skeleton*, scaled up until it broke ImageNet wide open — top-5 error dropped from ~26% to ~15% in a single year, a margin so large the field noticed overnight. What actually changed wasn't the recipe, it was everything *around* the recipe: ReLU instead of tanh (Concept 3 — no more vanishing gradients from saturation), training on two GPUs in parallel because no single GPU had enough memory, dropout (Concept 26, coming soon) to fight overfitting on a network with 60M parameters, and — critically — 1.2 million labeled ImageNet images instead of 60K MNIST digits. LeNet had the right idea in the wrong era. AlexNet had the same idea in the era that could finally feed it.

### The architectures, side by side

LeNet-5: input $32\times32\times1$ → conv($5\times5$, 6) → pool → conv($5\times5$, 16) → pool → conv($5\times5$, 120) → fc(84) → fc(10).

AlexNet: input $227\times227\times3$ → conv($11\times11$, stride 4, 96) → pool → conv($5\times5$, 256) → pool → conv($3\times3$, 384) → conv($3\times3$, 384) → conv($3\times3$, 256) → pool → fc(4096) → fc(4096) → fc(1000).

Notice the *shape* of the design is identical — a shrinking spatial pyramid of convs+pools funneling into a couple of fully-connected layers before the classifier. The parameter count for a conv layer is:

$$P_{\text{conv}} = (k \times k \times C_{in} + 1) \times C_{out}$$

where $k$ is kernel size and the $+1$ is the per-filter bias. Plug in AlexNet's first layer ($k{=}11, C_{in}{=}3, C_{out}{=}96$) and you get $(11 \cdot 11 \cdot 3 + 1) \times 96 \approx 34{,}944$ params doing $55\times55\times96$ worth of work — cheap per-parameter, expensive in compute, which is exactly why that layer used stride 4: an aggressive stride is a cheap way to shrink the spatial map fast when your input is huge.

Right: the derivative of tanh saturates toward zero away from the origin, while ReLU's derivative is a constant 1 for any positive input — the single biggest reason AlexNet trained fast enough to be practical.](graph_DAY_39.png)

### Why it matters / where it leads

This is the fork in the road for the entire rest of Phase 4. Every architecture from here — VGG (Concept 39, "what if we just go deeper with only 3×3s"), Inception (40, "what if branches at different kernel sizes run in parallel"), ResNet (41, "what if depth alone doesn't work past ~20 layers, here's why"), DenseNet (42) — is a direct argument *with* AlexNet's design choices. AlexNet proved scale works; the next decade of architecture research is entirely about finding cheaper, more stable ways to keep scaling.

**Real interview question:** "AlexNet used 11×11 filters with stride 4 in its first layer. VGG (a few years later) replaced every large filter with stacks of 3×3s and never went back. What's the receptive-field argument for why two stacked 3×3 convs can match one 5×5's receptive field, and what do you gain by paying for it in two smaller convs instead of one big one?"

## 🐍 PYTHONIC EDGE

Defining a multi-layer CNN like AlexNet in PyTorch, you'll constantly choose between `nn.Sequential` and a plain Python `list` of layers. One of these silently breaks; the other doesn't. This is a real bug people ship.

```python
import torch
import torch.nn as nn

torch.manual_seed(42)   # seeds this process's RNG once, unlike C++'s per-call srand pattern

# --- BAD: a plain Python list "looks" like it holds layers, but PyTorch can't see them ---
class BrokenAlexNetStem(nn.Module):                 # class X(Base): Python's single-inheritance
                                                       # syntax; C++ would write "class X : public Base"
    def __init__(self):
        super().__init__()                            # parent ctor call -- C++ instead uses an
                                                         # initializer list: BrokenAlexNetStem() : Base() {}
        self.layers = [                                # a plain list -- nn.Module has NO IDEA these
            nn.Conv2d(3, 96, kernel_size=11, stride=4),  # exist; .parameters() will skip them entirely
            nn.ReLU(),
            nn.Conv2d(96, 256, kernel_size=5, padding=2),
            nn.ReLU(),
        ]                                               # kwargs like kernel_size=, stride= are keyword-only
                                                          # by convention here, unlike positional-only C++ args

    def forward(self, x):
        for layer in self.layers:                       # a for-in loop over a Python list of callables --
            x = layer(x)                                  # first-class functions/objects, no function-pointer
                                                            # boilerplate needed like in C++
        return x

bad = BrokenAlexNetStem()
n_bad_params = sum(p.numel() for p in bad.parameters())   # generator expression inside sum() --
                                                             # no C++ equivalent without writing a loop
print("params PyTorch can see in the BAD version:", n_bad_params)   # -> 0. bad.to("cuda") would also
                                                                       #    silently leave these layers on CPU.

# --- GOOD: nn.ModuleList (or nn.Sequential) registers each submodule with the parent ---
class AlexNetStem(nn.Module):
    def __init__(self):
        super().__init__()
        self.layers = nn.ModuleList([                    # nn.ModuleList: a list-like container that IS
            nn.Conv2d(3, 96, kernel_size=11, stride=4),     # tracked -- __setattr__ on nn.Module registers
            nn.ReLU(),                                       # any nn.Module (or ModuleList/Sequential) value
            nn.Conv2d(96, 256, kernel_size=5, padding=2),
            nn.ReLU(),
        ])

    def forward(self, x):
        for layer in self.layers:
            x = layer(x)                                   # calling layer(x) invokes __call__, which wraps
        return x                                             # forward() with hooks -- never call .forward() directly

good = AlexNetStem()
n_good_params = sum(p.numel() for p in good.parameters())
print("params PyTorch can see in the GOOD version:", n_good_params)   # -> 269152 (correctly registered)

assert n_bad_params == 0 and n_good_params > 0
```

The fix costs nothing (`list` → `nn.ModuleList`) but the failure mode is brutal: the model trains, loss even goes down a little (from whatever layers *did* get registered), and you only find out your first two conv layers were frozen dead weight when accuracy plateaus far below what the architecture should hit.

## 📡 SIGNAL LAB

**Setup:** AlexNet's first layer is an $11\times11$ conv with **stride 4** applied directly to the raw $227\times227$ image. Treat that stride-4 conv as a low-pass filter (the $11\times11$ kernel, roughly box-like when untrained) followed by subsampling every 4th pixel. Question: does this operation risk aliasing, and does the $11\times11$ kernel's spatial support save it?

**Worked solution:** Subsampling by 4 drops the effective sample rate to $f_s/4$, so the new Nyquist frequency is $f_s/8$ (in original-image cycles/pixel units). For alias-free subsampling, the signal reaching the subsampler must have no energy above $f_s/8$. An $11\times11$ box-like kernel has a $\text{sinc}$-shaped frequency response with its first zero around $f \approx f_s/11$ — close to, but not comfortably below, $f_s/8$. Its response is *attenuated* well above $f_s/8$ but not zeroed until $f_s/11$, and sinc's slow-decaying sidelobes leak energy back in above that. Natural images also carry real high-frequency content (edges, fine texture) all the way up near $f_s/2$ that a single wide-but-shallow conv doesn't fully suppress. So yes — AlexNet's stem is a textbook mild aliasing source: some high-frequency image content folds down into lower apparent frequencies in the 55×55 output map.

**So what:** In classification, this barely matters — a few dB of aliased energy doesn't move top-5 accuracy, and the network's next layers just learn around it. But it's exactly the mechanism you should be suspicious of in your own research lane: any strided conv (not just the first layer — every downsampling stage in a CNN, and every stride in a GAN/diffusion decoder) is a subsampler preceded by an imperfectly-flat lowpass. That's a large piece of *why* GAN- and diffusion-generated images carry detectable periodic spectral artifacts (Concept 88, later this phase) — the up/downsampling stages in the generator alias in a signature, structured way that real camera sensors (with actual anti-aliasing optics in front of the sensor) mostly don't. Filed away for when we get there: "does this synthesis pipeline have a strided conv without adequate anti-aliasing before it?" is a legitimate first question to ask of any generative architecture.

## 🏋️ THE GAUNTLET

**LRU Cache**

Design a data structure for a Least Recently Used (LRU) cache. Implement:
1. `LRUCache(int capacity)` — initialize with positive size `capacity`.
2. `int get(int key)` — return the value if `key` exists, else `-1`. Counts as a "use" of that key.
3. `void put(int key, int value)` — insert or update the value. If inserting exceeds `capacity`, evict the *least recently used* key first.

Both `get` and `put` must run in **O(1)** average time.

**Constraints:** $1 \le \text{capacity} \le 3000$, $0 \le \text{key, value} \le 10^4$, up to $2 \times 10^5$ calls total.

*(Yes, this is the exact kind of memory-management problem the original AlexNet team faced for real — their 3GB-per-GPU limit is why the network was split across two GPUs with cross-connections only at specific layers. Here you're managing a fixed-capacity resource under the same "evict the least-recently-touched thing" pressure, just in code instead of silicon.)*

**Hints (escalating):**
1. O(1) `get`/`put` by key screams hash map — but a hash map alone can't tell you *recency order*, and a plain array/vector can't delete-and-reinsert in O(1) from an arbitrary position.
2. You need two structures working together: a hash map from `key → node`, and a structure that maintains usage order where you can move an arbitrary element to the "most recent" end in O(1) *without shifting anything else*. What structure supports O(1) removal from the middle given a pointer to the node?
3. A doubly linked list gives O(1) removal-given-a-node and O(1) insertion at either end. Keep the list ordered from least-recent (head) to most-recent (tail). `get(key)`: look up the node via the hash map, unlink it, relink it at the tail, return its value. `put(key, value)`: if key exists, update value and move-to-tail like `get`; if not and you're at capacity, unlink+erase the head node (and its hash map entry) before inserting the new node at the tail. Use dummy head/tail sentinel nodes to avoid null-pointer edge cases at the boundaries.

**Pattern:** Hash map + doubly linked list (intrusive, O(1) splice). **Target complexity:** $O(1)$ per `get`/`put`, $O(\text{capacity})$ space.

## 🔓 GAUNTLET SOLUTION

```cpp
#include <unordered_map>
using namespace std;

class LRUCache {
public:
    LRUCache(int capacity) : cap(capacity) {
        head = new Node(-1, -1);
        tail = new Node(-1, -1);
        head->next = tail;
        tail->prev = head;
    }

    int get(int key) {
        auto it = map.find(key);
        if (it == map.end()) return -1;
        Node* node = it->second;
        unlink(node);
        insertAtTail(node);
        return node->val;
    }

    void put(int key, int value) {
        auto it = map.find(key);
        if (it != map.end()) {
            Node* node = it->second;
            node->val = value;
            unlink(node);
            insertAtTail(node);
            return;
        }
        if ((int)map.size() == cap) {
            Node* lru = head->next;
            unlink(lru);
            map.erase(lru->key);
            delete lru;
        }
        Node* node = new Node(key, value);
        insertAtTail(node);
        map[key] = node;
    }

private:
    struct Node {
        int key, val;
        Node* prev = nullptr;
        Node* next = nullptr;
        Node(int k, int v) : key(k), val(v) {}
    };

    int cap;
    Node* head;
    Node* tail;
    unordered_map<int, Node*> map;

    void unlink(Node* node) {
        node->prev->next = node->next;
        node->next->prev = node->prev;
    }

    void insertAtTail(Node* node) {
        Node* last = tail->prev;
        last->next = node;
        node->prev = last;
        node->next = tail;
        tail->prev = node;
    }
};
```

`head` and `tail` are sentinel nodes so every real node always has a valid `prev`/`next` — no special-casing empty-list or single-element edge cases. `map` gives O(1) node lookup by key; `unlink` + `insertAtTail` give O(1) reordering given that node pointer. Every operation touches a fixed, small number of pointers regardless of cache size, satisfying the O(1) requirement.

## 💡 CONCEPT ANSWER

Receptive field after one $5\times5$ conv (stride 1) is $5\times5$. Two stacked $3\times3$ convs (stride 1) give receptive field $RF = 3 + (3-1) = 5$ — identical coverage. But the parameter/compute cost differs: one $5\times5$ over $C$ channels costs $25C^2$ params (ignoring bias), while two $3\times3$ layers cost $9C^2 + 9C^2 = 18C^2$ — fewer parameters for the same receptive field. More importantly, two stacked convs interleave **two ReLU nonlinearities** instead of one, so the effective function the network can represent between those input pixels and that output unit is strictly more expressive (a deeper composition of nonlinear maps) for less compute than the single wide conv. That's the exact argument VGG makes explicit and generalizes across its entire network: replace every large kernel with a stack of $3\times3$s, gain expressivity and save parameters simultaneously — the only cost is more sequential (less parallelizable) depth, and slightly more activation memory to store during the forward pass for backprop.
