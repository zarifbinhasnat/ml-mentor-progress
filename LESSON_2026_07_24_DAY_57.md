# Day 57 — Self- vs Cross-Attention

## 🧠 CONCEPT OF THE DAY

**Intuition.** Every attention block you've built since Day 53 answers one question: "for this query, how much should I weight each value?" Day 57's whole content is about *where the query comes from* versus *where the keys and values come from* — nothing else changes. **Self-attention**: a sequence looks at itself. Every token in a sentence asks "which other tokens in *this same sentence* are relevant to me?" — Q, K, and V are all projections of the identical input. **Cross-attention**: one sequence looks at a *different* sequence. A decoder token asks "which positions in *that other, already-encoded sequence* are relevant to me?" — Q comes from one place, K and V come from somewhere else entirely. Same scaled dot-product formula, same softmax, same `W^O` recombination from Day 56 — the only structural difference is whether the "who am I asking about" tensor and the "what am I asking" tensor share a source.

**The math.** Self-attention on a sequence `X ∈ ℝ^{n×d_model}`:

$$\text{SelfAttn}(X) = \text{softmax}\!\left(\frac{(XW^Q)(XW^K)^\top}{\sqrt{d_k}}\right) XW^V$$

Cross-attention between a query sequence `Y ∈ ℝ^{m×d_model}` (e.g. the decoder) and a context sequence `X ∈ ℝ^{n×d_model}` (e.g. the encoder output):

$$\text{CrossAttn}(Y, X) = \text{softmax}\!\left(\frac{(YW^Q)(XW^K)^\top}{\sqrt{d_k}}\right) XW^V$$

Notice the shapes: self-attention's score matrix is always `n×n` — square, because the query count equals the key count by construction. Cross-attention's score matrix is `m×n` — rectangular in general, because the decoder can have a completely different length than the encoder. That single shape fact is the whole distinction; everything else (the projections, the softmax, the weighted sum over `V`) is line-for-line identical code.

**Why it matters / where it leads.** This is literally the fix for Day 52's fixed-context bottleneck: the original seq2seq encoder–decoder (Day 51) compressed the *entire* source sentence into one fixed vector before the decoder ever saw it. Cross-attention lets every decoder step look back at *every* encoder position directly, weighted by relevance — no more single bottleneck vector. The original Transformer (Vaswani et al.) has exactly three attention blocks per layer: encoder self-attention, decoder *masked* self-attention (Day 62 preview), and decoder→encoder cross-attention. Here's the detail that trips people up in interviews: **modern decoder-only LLMs like GPT have no cross-attention at all** — they're pure causal self-attention stacks, because there's no separate "source sequence" to cross-attend to; the prompt and the generation live in the same stream. Cross-attention survives specifically in encoder–decoder architectures (T5, Whisper, the original Transformer) and — very relevant to your research lane — in text-to-image diffusion models, where the U-Net's cross-attention layers are literally how a Stable Diffusion model conditions image generation on a CLIP text embedding: image-latent patches are the queries, text-token embeddings are the keys/values.

**Interview-style question:** In an encoder–decoder Transformer, why must cross-attention recompute the encoder's K and V projections from the *encoder's final output* rather than reusing the encoder's own self-attention K/V from partway through the stack — and what does this imply about how much of the encoder must finish before decoding can start?

## 🐍 PYTHONIC EDGE

The bad way: write two separate classes, `SelfAttention` and `CrossAttention`, that duplicate 90% of their logic. The clean way: one class where "self" is just the special case of cross-attention where the context argument defaults to the input itself.

```python
import torch

class Attention(torch.nn.Module):
    # class X(Base): Python single-inheritance; C++ writes class X : public Base
    def __init__(self, d_model, h):
        super().__init__()                 # parent ctor call; C++ uses initializer-list syntax instead
        assert d_model % h == 0            # assert raises AssertionError if false — a no-op in C++ release builds
        self.h, self.d_k = h, d_model // h # tuple unpacking: two assignments in one statement; // = floor division
        self.q_proj = torch.nn.Linear(d_model, d_model)
        self.k_proj = torch.nn.Linear(d_model, d_model)
        self.v_proj = torch.nn.Linear(d_model, d_model)
        self.out_proj = torch.nn.Linear(d_model, d_model)

    def _split_heads(self, t, b, n):
        # (b, n, d_model) -> (b, h, n, d_k); no C++ equivalent for chained .view()/.transpose() reshaping
        return t.view(b, n, self.h, self.d_k).transpose(1, 2)

    def forward(self, x, context=None):    # context=None is Python's default-argument mechanism (no overloading needed)
        if context is None:                # `is None` (identity check), not `== None` — idiomatic Python vs C++ nullptr check
            context = x                    # no context given -> self-attention: keys/values come from x itself
        b, n, d = x.shape                  # tuple unpacking; .shape is a tuple-like object, not a method call
        m = context.shape[1]               # context's sequence length may differ from x's (cross-attention case)

        q = self._split_heads(self.q_proj(x), b, n)         # calling a module invokes __call__ -> wraps forward()
        k = self._split_heads(self.k_proj(context), b, m)   # projected from context, not x — the whole distinction
        v = self._split_heads(self.v_proj(context), b, m)

        out = torch.nn.functional.scaled_dot_product_attention(q, k, v)  # batched matmul over (b, h) automatically
        out = out.transpose(1, 2).contiguous().view(b, n, d)  # .contiguous() mandatory after .transpose() before .view()
        return self.out_proj(out)

attn = Attention(d_model=64, h=8)
decoder_tokens = torch.randn(2, 5, 64)   # (batch, decoder_len, d_model)
encoder_tokens = torch.randn(2, 9, 64)   # (batch, encoder_len, d_model) — different length, no problem

self_out = attn(decoder_tokens)                       # context defaults to decoder_tokens -> self-attention
cross_out = attn(decoder_tokens, context=encoder_tokens)  # keyword arg -> cross-attention, m != n is fine
```

The bug this idiom prevents: if you'd hand-written two classes, it's easy for `SelfAttention` and `CrossAttention` to drift apart during refactors (one gets a bugfix, the other doesn't). One class with an optional `context` argument makes "self-attention" a *call-site* decision, not an *architectural* one — exactly how PyTorch's own `nn.MultiheadAttention(query, key, value)` is designed (pass `query=key=value=x` for self-attention, distinct tensors for cross-attention).

## 📡 SIGNAL LAB

Self- vs cross-attention has an exact DSP twin: **autocorrelation vs cross-correlation**. Autocorrelation slides a signal against a *delayed copy of itself* to find internal structure — periodicity, repeated patterns. Cross-correlation slides one signal against a *different* signal to find how they relate — the lag that best aligns them, used in everything from TDOA (time-difference-of-arrival) localization to template matching. Self-attention finding "which of my own tokens matter to me" is exactly autocorrelation finding "which of my own past samples predict me"; cross-attention finding "which encoder tokens matter to this decoder step" is exactly cross-correlation finding "how does this other signal align with mine."

**Worked check:**

```python
import numpy as np
np.random.seed(42)

fs = 256                                            # sample rate (Hz)
t = np.arange(0, 1, 1 / fs)
period_hz = 10
x = np.sin(2 * np.pi * period_hz * t) + 0.3 * np.random.randn(len(t))   # noisy periodic signal

# ---- self case: autocorrelation finds x's OWN internal periodic structure ----
def autocorr(sig):
    sig = sig - sig.mean()
    r = np.correlate(sig, sig, mode="full")         # full sliding correlation of sig against itself
    return r[r.size // 2:]                          # keep non-negative lags only

ac = autocorr(x)
lag_peak = np.argmax(ac[5:]) + 5                    # skip the trivial zero-lag peak
print(f"autocorrelation peak lag: {lag_peak} samples -> period {lag_peak / fs * 1000:.1f} ms "
      f"(true period {1000 / period_hz:.1f} ms)")

# ---- cross case: cross-correlation finds the alignment between x and a DIFFERENT signal y ----
true_delay = 17
y = np.roll(x, true_delay) + 0.1 * np.random.randn(len(x))   # a related but genuinely different signal

def crosscorr(a, b):
    a, b = a - a.mean(), b - b.mean()               # tuple unpacking: two assigns in one statement
    r = np.correlate(a, b, mode="full")
    lags = np.arange(-len(a) + 1, len(a))
    return lags, r

lags, cc = crosscorr(y, x)
est_delay = lags[np.argmax(cc)]
print(f"cross-correlation estimated delay: {est_delay} samples (true delay: {true_delay})")
```

Autocorrelation's peak lag recovers `x`'s own period (~25-26 samples ≈ 100 ms, matching the 10 Hz tone) purely from `x` compared against itself — no second signal needed, just like self-attention needs no second sequence. Cross-correlation's peak lag recovers the *known* 17-sample delay between two genuinely different signals — just like cross-attention recovers *alignment* between decoder and encoder positions, except attention learns a soft, content-dependent alignment (a full weighted distribution over all encoder positions per query) rather than a single hard best-lag scalar. **So what:** the same underlying operation — slide-and-correlate — answers "what's my own structure?" when applied to one signal, and "how do these two signals relate?" when applied to two; self- vs cross-attention is that exact duality lifted into learned, content-adaptive, multi-head form.

## 🏋️ THE GAUNTLET

**Problem: Longest Common Subsequence**

Given two strings `text1` and `text2`, return the length of their longest common subsequence (not necessarily contiguous — think of it as finding the best soft *alignment* between two different sequences, the classical-algorithms cousin of what cross-attention learns implicitly).

**Constraints:** `1 ≤ text1.length, text2.length ≤ 1000`, both strings consist of lowercase English letters.

**Target:** `O(n·m)` time, `O(min(n, m))` extra space.

**Hints (use only if stuck, in order):**
1. Define `dp[i][j]` = length of the LCS of `text1[:i]` and `text2[:j]`. What's the recurrence when `text1[i-1] == text2[j-1]` versus when they differ?
2. Base case: `dp[0][j] = dp[i][0] = 0` for all `i, j` (an empty string has no common subsequence with anything). If characters match, you extend a diagonal-predecessor match; if they don't, you take the best of dropping one character from either string.
3. You never need more than the *previous row* of the DP table to compute the current row — collapse the 2D table to two 1D rolling arrays (or one array updated carefully in the right direction) to hit `O(min(n, m))` space.

**Pattern:** Two-sequence alignment DP — recognize this whenever you need the best correspondence between two *different* sequences (diff tools, edit distance, bioinformatics sequence alignment, and — thematically — the classical, hard-alignment ancestor of what cross-attention's soft weights approximate).

## 🏗️ BLUEPRINT

**Cross-attention forces a synchronization point that self-attention-only (decoder-only) architectures don't have.** In an encoder–decoder model, every decoder layer's cross-attention needs the *encoder's finished* K/V projections — which means the entire encoder stack must run to completion before decoding can begin, and those K/V tensors (size `O(n · d_model)` per layer) must stay resident in memory for the whole generation, since every future decode step re-attends to the same encoder output. A decoder-only model has no such barrier: there's nothing to wait on before the first token, and there's no separate encoder K/V cache to keep alive — just the growing self-attention KV cache (Day 100 preview) of its own generated tokens. This is one of the practical reasons decoder-only architectures (GPT-style) dominate general-purpose LLM serving over encoder–decoder ones (T5-style): one fewer stage of sequential dependency and one fewer resident cache to manage at inference time, even though encoder–decoder models can be more compute-efficient per-token for tasks with a clear, fixed "input" (translation, summarization) since the encoder only has to run once regardless of how many tokens the decoder later generates.

🔓 GAUNTLET SOLUTION

```cpp
class Solution {
public:
    int longestCommonSubsequence(string text1, string text2) {
        int n = text1.size(), m = text2.size();
        // ensure text2 (inner loop dim) is the shorter string to minimize space
        if (m > n) swap(text1, text2), swap(n, m);

        vector<int> prev(m + 1, 0), curr(m + 1, 0);

        for (int i = 1; i <= n; i++) {
            for (int j = 1; j <= m; j++) {
                if (text1[i - 1] == text2[j - 1]) {
                    curr[j] = prev[j - 1] + 1;
                } else {
                    curr[j] = max(prev[j], curr[j - 1]);
                }
            }
            swap(prev, curr);   // curr becomes next iteration's prev; O(1) swap instead of O(m) copy
        }

        return prev[m];
    }
};
```

Each of the `n·m` cells is computed once in `O(1)`: `O(n·m)` time. Only two rolling rows of size `m+1` are kept alive at once: `O(min(n, m))` space after putting the shorter string on the inner dimension.

💡 CONCEPT ANSWER

Because self-attention's K/V at any encoder layer are computed from that layer's *own* intermediate representation, and each encoder layer refines that representation further — the K/V a decoder would get from, say, layer 3 of a 6-layer encoder simply don't reflect the context that layers 4–6 would still add. Cross-attention needs to attend to the encoder's *best available understanding* of the source sequence, which only exists once the full encoder stack has finished processing. This has a real systems consequence: encoder–decoder architectures have a hard sequential barrier — the entire encoder forward pass must complete before the first decoder token can be produced, unlike decoder-only architectures where generation can begin immediately since there's no separate "other sequence" to wait on. It's also why, once computed, the encoder's final K/V are cached and reused unchanged across every decoding step rather than recomputed — they're a fixed reference the whole generation attends back into.
